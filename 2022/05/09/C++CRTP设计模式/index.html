C++ CRTP设计模式

发表于 2022-05-09 更新于 2022-09-07 分类于 C++ 阅读次数： Valine：

概述

将Eigen中广泛使用的CRTP设计模式的内容总结一下

背景

C++ 通过类的继承与虚函数的动态绑定，实现了（动态）多态


在每次执行f(int i)的时候，系统会检查b指向的实例实际的类型，然后调用对应类型的f函数。这一步骤需要通过查询虚函数表（vtable）来实现；由于实际b指向对象的类型在运行时才确定（而不是在编译时就确定），所以这种方式称为动态绑定（或者运行时绑定）。
因为每次都需要查询虚函数表，所以动态绑定会降低程序的执行效率。为了兼顾多态与效率，有人提出了Curiously Recurring Template Pattern （奇异递归模板模式）的概念。

特点

  • 基类是一个模板类，接收子类的类型名字，因此子类的继承列表会类似于Derived: public Base<Derived>；
  • 基类的函数在函数体中，使用 static_cast< Derived* > 将基类的指针转为（模板）子类的指针，在编译期完成绑定，因此CRTP设计模式也称为静态多态，也可以叫编译期多态

Eigen实例

核心点有两个：

  • Eigen的类层次结构旨在避免虚函数，因为它们的开销会显着影响性能。相反，Eigen 使用CRTP实现了静态多态

  • Eigen的类设计避免多继承，因为发现有的编译器不支持空基类优化（EBO），而EBO这对于Eigen的固定大小类型至关重要

下面以Eigen中的Matrix类的实现来说明CRTP的实现

Matrix以及Array的类继承体系如下图所示，直接基类PlainObjectBase和Matrix使用了CRTP模式，Matrix将自己作为参数传入到继承的基类中。而其他的基类的继承采用的是普通的继承关系。

看一下EigenBase中的一些接口，发现cols()，rows()这些是通过派生类间接实现的，那么在派生类中就不用再次定义这些接口了。基于此，我们发现Matrix中的接口集中是一些构造接口，并非是预想的那样，里面会有好多矩阵操作的接口。实际上，Eigen正是将矩阵的各种操作进行不断的抽象，才出现我们今天看到的这么多继承等级。除此之外，EigenBase等也可以单独当作模板使用。
